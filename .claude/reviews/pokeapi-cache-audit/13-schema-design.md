# 13 — Schema-design / best-practices review

Findings from reviewing all 18 schema files in [database/schemas/pokeapi/](../../../database/schemas/pokeapi/), [columnHelpers.ts](../../../database/utils/columnHelpers.ts), [seed.ts](../../../database/seeds/pokeapi/seed.ts), and the queries in [app/models/pokeapi.server.ts](../../../app/models/pokeapi.server.ts).

## Primary keys

**Pattern:** Reference data (types, regions, locations, generations, languages, pokemon, pokemon_species, pokedex, abilities, moves, items, berries, machines, characteristics, natures, etc.) uses **PokeAPI integer IDs as primary keys**. Junction/name/effect tables use **autogenerated surrogate IDs** (`.generatedAlwaysAsIdentity()`).

**Assessment:** consistent and justified. PokeAPI IDs are stable across cache refreshes; using them as PKs removes the need for an extra lookup column. The risk (PokeAPI renumbering) is theoretical.

**One inconsistency:** `evolutionChainsTable.evolutionChainId` is `integer().primaryKey()` (line 185) — uses PokeAPI ID. `pokemonSpeciesEvolutionsTable.pokemonSpeciesEvolutionId` is a surrogate (line 216-218). Pattern is correct — chain has a stable PokeAPI ID, individual evolution edges don't.

## Naming conventions

**Table names:** all snake_case in DB, all camelCase + `Table` suffix in TS — fully consistent. Plural for collection tables (`pokemon`, `abilities`, `types`), singular for the "one row per type" semantic table (`pokemon` is treated as the collection, no `_s` because already plural).

**Outliers:**
- `pokemonTable` exports as plain `pokemon` (no `s`); all other entity tables use plural. Acceptable because "pokemon" is already plural in English.
- `typeEfficacyTable` uses `damageFactor` (integer percentage 0/25/50/100/200) — undocumented in column comment beyond the line 42 comment. Worth a CHECK constraint to enforce valid values.

## Indexes

**Existing:** every junction/name table has a `uniqueIndex` on its natural key tuple (`(parentId, languageId)`, `(pokemonId, slot)`, etc.). This is consistently applied.

**Missing indexes (lookup performance):**
- `pokemonTable.pokemonSpeciesId` — heavily joined in [pokeapi.server.ts:248-251](../../../app/models/pokeapi.server.ts) (`getPokemonDetailsByName` `fullJoin`). Missing index → seq scan on every lookup.
- `pokemonSpeciesTable.evolvesFromSpeciesId` — joined in [pokeapi.server.ts:192-198](../../../app/models/pokeapi.server.ts).
- `pokemonSpeciesTable.evolutionChainId` — joined in [pokeapi.server.ts:185-191](../../../app/models/pokeapi.server.ts).
- `pokemonFormsTable.pokeApiId` (the FK to pokemon) — likely queried by form lookup.
- `pokemonSpritesTable.pokemonId` — already covered by the unique `(pokemonId, source, variant)` index (leftmost column rule), so query by `pokemonId` alone uses the index. ✓
- `pokemonSpeciesEvolutionsTable` — no semantic index. Surrogate PK only. Queries by `evolutionChainId` or `evolveStartSpeciesId` will seq-scan.

**Recommendation:** Add B-tree indexes on the FK columns enumerated above. Composite uniques on junction tables are fine but PKs of `id` only mean FK lookups need their own index.

## Foreign keys

**`.references()` usage is consistent for most tables**, but there are documented and undocumented holes:

1. **Documented implicit FKs (cross-file circular import avoidance):**
   - `pokemonSpeciesVarietiesTable.pokemonId` ([pokemon-species.ts:199](../../../database/schemas/pokeapi/pokemon-species.ts)) — comment explains. Could be resolved by moving `varieties` to pokemon.ts or using a separate junction file.
   - `pokemonSpeciesEvolutionsTable.evolveStartPokemonId`, `evolveEndPokemonId` ([pokemon-species.ts:226, 231](../../../database/schemas/pokeapi/pokemon-species.ts)) — same.

2. **Undocumented missing FKs (bugs to fix):**
   - **`pokemonSpeciesTable.evolvesFromSpeciesId`** ([pokemon-species.ts:64](../../../database/schemas/pokeapi/pokemon-species.ts)) — plain `integer()`. Should be `.references(() => pokemonSpeciesTable.pokemonSpeciesId)` (self-reference works in Drizzle).
   - **`pokemonSpeciesTable.evolutionChainId`** ([pokemon-species.ts:65](../../../database/schemas/pokeapi/pokemon-species.ts)) — plain `integer()`. Should `.references(() => evolutionChainsTable.evolutionChainId)` — currently relies on the backfill at [seed.ts:2393-2409](../../../database/seeds/pokeapi/seed.ts) being correct.

3. **`pokemonSpeciesEvolutionsTable.genderId`** ([pokemon-species.ts:249](../../../database/schemas/pokeapi/pokemon-species.ts)) DOES have `.references(gendersTable)`, but the seed (`collectEvolutions` line 1990) inserts the raw `detail.gender` int directly without verifying it's a valid gender ID. If PokeAPI ever returns a gender value not in the gendersTable, the insert will fail at FK enforcement. Acceptable.

## Constraints

**`.notNull()` is reasonable** for most scalar fields. A few worth double-checking:
- `pokemonFormsTable.formIdentifier` is `notNull()` but empty string for default forms (per Batch 10 finding). Consider making nullable.
- `pokemonTable.baseExperience` is nullable (line 29) — correct, some pokemon (e.g. eternatus 890) have null base_experience.

**CHECK constraints are not used anywhere.** Candidates:
- `typeEfficacyTable.damageFactor` IN (0, 25, 50, 100, 200) — only 4 valid values.
- `pokemonAbilitiesTable.slot`, `pokemonTypesTable.slot` IN (1, 2, 3) — slot values are small enumerated.
- `pokemonSpeciesTable.genderRate` BETWEEN -1 AND 8 — PokeAPI uses -1 for genderless, 0–8 for ratio.
- `pokemonSpeciesEvolutionsTable.genderId` BETWEEN 1 AND 3 if treated as an enum.

These are nice-to-have, not bugs.

**Unique indexes:** all junction tables enforce semantic uniqueness via uniqueIndex. ✓

## Soft delete

**Issue:** the `timestamps` helper ([columnHelpers.ts:4-10](../../../database/utils/columnHelpers.ts)) includes `deletedAt` for EVERY table, including pure reference data (types, languages, generations, regions, stats, natures, etc.). For these, deletion is not a meaningful operation — they're seeded from external data.

**Recommendation:** split `timestamps` into:
- `timestamps` (createdAt + updatedAt only — for reference data)
- `softDeletableTimestamps` (adds deletedAt — for app-owned data like trainers, pokemon instances, teams)

This avoids carrying useless columns on 135 pokeapi tables.

## Junction tables

All M2M tables use surrogate PKs + unique indexes on the natural key. Pattern is consistent:
- `pokemonAbilitiesTable`: `pokemonAbilityId` (surrogate) + unique (`pokemonId, slot`)
- `typeEfficacyTable`: `typeEfficacyId` (surrogate) + unique (`attackingTypeId, defendingTypeId`)
- etc.

**Question:** could these be composite PKs instead of surrogate + unique? Saves an int and an index. The current pattern is "always surrogate" which is consistent and Drizzle-friendly — accept as is.

## Naming inconsistencies / duplications

**`*Names` table proliferation:** every entity has its own `*Names` table — `typeNamesTable`, `abilityNamesTable`, `moveNamesTable`, `regionNamesTable`, `pokemonSpeciesNamesTable`, etc. ~25+ near-identical tables.

**Tradeoff:**
- **Current (per-entity):** clean FKs, type-safe queries, each i18n table is just `(parentId, languageId, name)`.
- **Polymorphic translations table:** one `translationsTable` with `(entityType, entityId, languageId, name)`. Saves table count but loses FK enforcement and needs application-level discriminator handling.

**Recommendation:** keep the per-entity approach. PostgreSQL handles 100+ tables fine; the FK safety is worth it.

## Type widths

`varchar(255)` is the default for names/identifiers. A few outliers:
- `pokemonSpritesTable.url` is `varchar(500)` (line 240) — appropriate, sprite URLs are longer than entity URLs.
- `pokemonSpeciesEvolutionsTable.timeOfDay` is `varchar(50)` (line 261) — fine, values are "day"/"night"/"" (normalized to null).
- `pokemonSpeciesEvolutionsTable.baseForm` is `varchar(255)` (line 262) — fine.
- `text` used in `growthRatesTable.formula` (line 74), `pokemonSpeciesFlavorTextsTable.flavorText` (line 106). Appropriate.

**One concern:** `varchar(255)` for URLs everywhere — long URLs (e.g. some sprite paths) could exceed this. Consider `text` for URL columns or `varchar(500)`.

## `onConflictDoNothing` in seed

**Pattern in seed.ts:**
- Parent inserts (entities with PokeAPI ID as PK) often use `.onConflictDoNothing()` — idempotent re-seeds.
- Child inserts (junction tables with surrogate PK) use plain `insertChunked` — re-seed will produce duplicates or error on the unique index.

**Risk:** if seed is run twice without truncating, child tables either error (good — surfaces the issue) or silently duplicate (bad — only if the unique index is missing, which Batch 13 confirms shouldn't happen).

**Recommendation:** the test pattern truncates before seed via `truncateAllTables()`. For production, document the assumption that seed runs against an empty schema OR add `.onConflictDoNothing()` to all child inserts.

## Drizzle relations()

[database/schema.ts](../../../database/schema.ts) does not declare any `relations()`. The queries in [app/models/pokeapi.server.ts](../../../app/models/pokeapi.server.ts) use explicit `.leftJoin(...).on(eq(...))` instead of the relational query API.

**Tradeoff:**
- Without `relations()`: more verbose queries, but explicit and transparent.
- With `relations()`: enables `db.query.pokemonSpeciesTable.findFirst({ with: { evolutionChain: { with: { evolutions: true } } } })`-style queries, which would be much terser for the multi-table joins in `getAllPokemonSpeciesDataById`.

**Recommendation:** add `relations()` declarations for at least the heavy-join paths (pokemonSpecies → evolutionChain → evolutions; pokemon → species → varieties). Worth ~50 lines of one-time setup for cleaner queries.

## Top recommendations (prioritized)

1. **[high]** Add missing FK indexes (esp. `pokemonTable.pokemonSpeciesId`, `pokemonSpeciesTable.evolvesFromSpeciesId`, `pokemonSpeciesTable.evolutionChainId`, `pokemonFormsTable.pokeApiId`) — direct query performance impact.
2. **[high]** Add `.references()` to `pokemonSpeciesTable.evolvesFromSpeciesId` (self-ref) and `.evolutionChainId` — these are real bugs masquerading as "implicit FKs" because the schema's intent is documented for the varieties/evolutions tables but not these. Currently relies on backfill correctness.
3. **[medium]** Split `timestamps` helper — `deletedAt` doesn't belong on pure reference data (~135 columns of waste).
4. **[medium]** Add `relations()` to schema.ts for the heavy-join paths used in [app/models/pokeapi.server.ts](../../../app/models/pokeapi.server.ts). Enables query-builder style and lets removed code (commented-out `getEvolutionChain` at line 300-380) be reintroduced cleanly.
5. **[medium]** Add a `uniqueIndex` to `pokemonSpeciesEvolutionsTable` on the semantic key (e.g. `(evolutionChainId, evolveStartSpeciesId, evolveEndSpeciesId, triggerId)` plus the disambiguating conditions). Currently re-seed without truncation would duplicate rows.
6. **[low]** Make `pokemonFormsTable.formIdentifier` nullable (currently NOT NULL with empty strings for default forms).
7. **[low]** Add `.onConflictDoNothing()` to all child-table inserts in seed for production idempotency (or document the truncate-first assumption explicitly).
8. **[low]** Add CHECK constraints on `typeEfficacyTable.damageFactor` (∈ {0, 25, 50, 100, 200}) and other small-enum columns.
9. **[low]** Widen URL columns to `varchar(500)` or `text` consistently — currently mixed widths.
